## 2024-04-25 - Analytics Database Query Overfetching Anti-Pattern
**Learning:** Found an anti-pattern in `app/api/analytics/*` routes where `new PrismaClient()` is improperly instantiated and `findMany()` is used to fetch all records into Node.js memory just for aggregate counts, leading to potential memory bloat, high latency, and DB connection exhaustion.
**Action:** Always import the shared singleton `import { prisma } from '@/lib/prisma';`. Use database-level aggregations like `prisma.parcel.groupBy()` with `_count: { _all: true }` and accumulate mapped fallback keys in memory to minimize database transfer latency and Node.js memory footprint.
## 2024-05-18 - Replacing N+1 Overfetching with Database Aggregations
**Learning:** In Prisma, eagerly fetching deeply nested relations using `include` within list endpoints (like `app/api/contractors/route.ts`) pulls large JSON payloads into Node.js memory. This leads to heavy memory usage, slow serialization, and bottlenecks when performing array length/aggregation calculations via `.map` or `.reduce`.
**Action:** Always prefer pushing aggregate calculations to the database using `include: { _count: { ... } }` for relation lengths, and `.groupBy` with `_avg`/`_sum` for deeper numerical aggregations. These approaches condense an O(N*M) in-memory operation into a single O(1) database round-trip, significantly optimizing the API response payload and backend server memory.
