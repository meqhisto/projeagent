## 2024-05-30 - Fix N+1 Query in Contractors List via Prisma Aggregation
**Learning:** Fetching nested arrays containing full relations (e.g. `include: { ratings: true, matches: { include: { parcel: true, customer: true } } }`) just to count them or perform local mathematical aggregations on a list view introduces significant database roundtrips, large payloads, and high memory usage (N+1-like issue).
**Action:** When needing counts or aggregated computations on related fields in list views, avoid over-fetching by using Prisma's `_count` and pushing computations to the database using `groupBy` (e.g., `_avg`). Extract specific keys with `.map()` and batch these aggregate lookups using `{ where: { field: { in: ids } } }`, then map back via a JS `Map` object for O(1) correlation in memory.
