## 2024-04-22 - Replacing Memory-Intensive Analytics Queries with Prisma Aggregations
**Learning:** The project was experiencing an anti-pattern in the analytics API endpoints (`app/api/analytics/**`) where the application instantiated a `new PrismaClient()` dynamically for each request and used `findMany()` to fetch entire tables of data just to calculate basic length, count, sum, or group aggregates in Node.js memory. This causes massive memory spikes, long TTFB times, N+1 query effects, and runs the risk of exhausting DB connection limits via multiple instantiated Prisma clients.
**Action:** Always import the shared `prisma` singleton (`import { prisma } from "@/lib/prisma"`) instead of instantiating `new PrismaClient()`. In analytics routes or anywhere computing metrics, replace `findMany()` logic with database-level aggregations (`prisma.parcel.count()`, `prisma.parcel.aggregate()`, and `prisma.parcel.groupBy()`) and wrap independent calculations in `Promise.all` to run them concurrently.
