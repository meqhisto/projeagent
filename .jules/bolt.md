## 2024-03-24 - Database Over-Fetching & Connection Leaks in Next.js Serverless Routes
**Learning:** Found multiple serverless API routes (`app/api/analytics/*`) directly instantiating `new PrismaClient()` and fetching full database rows (e.g. `prisma.parcel.findMany()`) into memory just to map and count/aggregate them in Node.js. This leads to two critical issues: 1) Connection pool exhaustion due to repeatedly instantiating new Prisma clients on every request instead of reusing the shared singleton (`@/lib/prisma`), and 2) Memory bloat and slow latency from transferring unused data from PostgreSQL/SQLite to the Node server when aggregations could be handled entirely by the database.
**Action:** Always import the shared `import { prisma } from '@/lib/prisma'` instance. For analytics and KPIs, prioritize database-level aggregations (`.count()`, `.aggregate()`, `.groupBy()`) via `Promise.all` over fetching array lists into memory, except when mapping complex date operations where a constrained `.findMany({ select: { id: true, date: true } })` is more efficient than forcing database dialect-specific grouping queries.
