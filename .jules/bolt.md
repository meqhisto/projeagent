## 2024-04-25 - Analytics Database Query Overfetching Anti-Pattern
**Learning:** Found an anti-pattern in `app/api/analytics/*` routes where `new PrismaClient()` is improperly instantiated and `findMany()` is used to fetch all records into Node.js memory just for aggregate counts, leading to potential memory bloat, high latency, and DB connection exhaustion.
**Action:** Always import the shared singleton `import { prisma } from '@/lib/prisma';`. Use database-level aggregations like `prisma.parcel.groupBy()` with `_count: { _all: true }` and accumulate mapped fallback keys in memory to minimize database transfer latency and Node.js memory footprint.
## 2024-05-19 - N+1 and Network Payload Optimization via _count and select in Prisma Lists
**Learning:** Returning fully populated relational properties (e.g. `matches` arrays) via `include` for list endpoints, only to calculate `.length` on the frontend, causes significant database over-fetching overhead. Additionally, fetching array relations purely for server-side score calculations but exposing them implicitly bloated JSON payload sizes over the network.
**Action:** Use Prisma's `_count` parameter inside `select` inside `findMany` queries for calculating relational counts instead of including whole sets. Explicitly limit intermediate attributes (like nested objects used solely for an average calculations) fetched using nested `select` and destructing to eliminate them before serializing the JSON response back to the client.
