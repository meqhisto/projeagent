## 2024-04-25 - Analytics Database Query Overfetching Anti-Pattern
**Learning:** Found an anti-pattern in `app/api/analytics/*` routes where `new PrismaClient()` is improperly instantiated and `findMany()` is used to fetch all records into Node.js memory just for aggregate counts, leading to potential memory bloat, high latency, and DB connection exhaustion.
**Action:** Always import the shared singleton `import { prisma } from '@/lib/prisma';`. Use database-level aggregations like `prisma.parcel.groupBy()` with `_count: { _all: true }` and accumulate mapped fallback keys in memory to minimize database transfer latency and Node.js memory footprint.
## 2025-02-18 - Reduce Overfetching by Replacing Nested Includes with Counts
**Learning:** In list views where only the length of a relation is needed (e.g., `matches.length` or `ratings.length`), using `include` to fetch full relational graphs causes massive database overfetching and memory overhead, especially when nested includes (e.g., `parcel` and `customer` within `matches`) are involved.
**Action:** Always use Prisma's `_count: { select: { relation: true } }` feature instead of fetching the entire array when only the count is needed. This reduces DB round trips, JSON serialization overhead, and network payload size. When returning data from API routes, explicitly destructure and omit arrays that are only fetched for intermediate server-side calculations (like average score).
