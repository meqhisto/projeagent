## 2024-04-25 - Analytics Database Query Overfetching Anti-Pattern
**Learning:** Found an anti-pattern in `app/api/analytics/*` routes where `new PrismaClient()` is improperly instantiated and `findMany()` is used to fetch all records into Node.js memory just for aggregate counts, leading to potential memory bloat, high latency, and DB connection exhaustion.
**Action:** Always import the shared singleton `import { prisma } from '@/lib/prisma';`. Use database-level aggregations like `prisma.parcel.groupBy()` with `_count: { _all: true }` and accumulate mapped fallback keys in memory to minimize database transfer latency and Node.js memory footprint.
## 2024-05-04 - Overfetching relational arrays just for counts and manual averages
**Learning:** In list endpoints like `app/api/contractors/route.ts`, deeply nested relational queries (e.g. `include: { ratings: true, matches: { include: ... } }`) were used purely to calculate lengths (`ratings.length`) and arithmetic averages in memory. This wastes significant memory and database transfer bandwidth.
**Action:** When computing relationship counts or aggregates across multiple records in a list view, use Prisma's `_count` aggregate inside the `include` block for simple lengths, and execute separate `prisma.<Model>.groupBy()` queries with aggregate functions (`_avg`, `_sum`, etc.) along with a `where` condition to push expensive processing directly to the database. Map the results back to the original objects in-memory.
