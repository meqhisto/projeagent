## 2024-03-27 - Stop DB-to-Node Memory Transfer for Aggregations
**Learning:** Found a systemic anti-pattern in the codebase where analytics routes (`app/api/analytics/kpis/route.ts`, `app/api/analytics/pipeline/route.ts`, `app/api/analytics/monthly-trend/route.ts`) were calling `prisma.parcel.findMany()` to fetch all parcel rows (sometimes even `include`ing relations) simply to count items, group them, or sum columns in Node.js using `.filter()`, `.reduce()`, and `.length`. This triggers massive database round-trips, extreme memory consumption in the Edge/Node.js worker, and DB connection exhaustion as some routes instantiated `new PrismaClient()` explicitly.
**Action:** Always prefer Prisma's `.count()`, `.aggregate({_sum})`, and `.groupBy()` APIs pushed directly to PostgreSQL. For concurrent independent aggregations, execute them efficiently using `Promise.all([prisma.count(), prisma.aggregate()])`. Also, when extracting analytics logic, always reuse the `import { prisma } from "@/lib/prisma"` singleton instead of creating local connections.
