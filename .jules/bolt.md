## 2024-05-18 - Database Aggregation over Memory Iteration
**Learning:** In Next.js API routes or server environments, calculating aggregate KPIs or summary metrics (like count, sum, average) by pulling all rows into Node.js memory using `findMany()` creates unnecessary bottlenecks, high RAM usage, and long request times as datasets grow. Additionally, when handling database enumerations or grouped values with null fallbacks (e.g., `item.crmStage || "NEW_LEAD"`), overwriting grouped results can cause data loss.
**Action:** Always prefer pushing processing down to the database layer by utilizing Prisma's `.count()`, `.aggregate()`, or `.groupBy()` methods with `_count: { _all: true }`. When mapping aggregate results that depend on default string fallbacks in Node.js, ensure identical fallback groups are accumulated (summed together) rather than overwritten. Ensure database aggregation optimization is considered immediately when implementing any data summarization endpoints.
