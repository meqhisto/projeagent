## 2024-05-18 - Optimized Prisma aggregation to push processing to the database
**Learning:** Found instances of memory-heavy calculations in the backend API using `findMany` combined with Node.js array methods (like `.filter().length`), leading to O(N) memory allocation to load large records into Node.js instead of utilizing efficient database-level `GROUP BY`. Also, directly instantiating `new PrismaClient()` in API routes causes database connection exhaustion in serverless Next.js environments.
**Action:** When calculating KPIs or data aggregates, always use Prisma's `.count()`, `.aggregate()`, or `.groupBy()` methods (pushing the query logic to the DB and accumulating values like `counts[stage] += g._count._all` to safely handle defaults) to avoid memory overhead. Furthermore, never instantiate `new PrismaClient()` directly; use the exported singleton instance from `@/lib/prisma`.
